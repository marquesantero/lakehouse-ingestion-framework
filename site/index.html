ContractForge

Framework batch declarativo para ingestão em Delta Lake no Databricks com contratos por tabela, arquitetura Medallion, quality gates, SCD, watermarks tipados, snapshot com soft delete, evolução de schema, explain mode e eventos OpenLineage.

Python ≥3.10 PySpark ≥3.4 Delta Lake ≥3.0 MIT v1.14.0 presets shape

O que é?

Uma biblioteca Python que encapsula padrões recorrentes de ingestão em Delta Lake com contratos declarativos. Em vez de escrever scripts ad-hoc com MERGE INTO, INSERT, OVERWRITE, você descreve o que fazer via um IngestionPlan (Python ou YAML), e o framework executa como fazer de forma padronizada, com observabilidade completa em tabelas de controle.

Escopo: não orquestra DAGs (use Databricks Workflows, Airflow ou DAB) e não faz streaming contínuo. A lib suporta Auto Loader em available_now para ingestão finita com checkpoint.

Arquitetura Medallion

CamadalayerModos típicosPropósito
Bronze"bronze"scd0_append, scd0_overwrite, scd1_hash_diffCaptura bruta, preservação
Silver"silver"scd1_upsert, scd2_historical, snapshot_soft_deleteQualidade, consolidação, histórico
Gold"gold"scd0_overwrite, scd1_upsertConsumo, agregações

A tabela alvo é montada como {catalog}.{layer}.{target_table}. Ex.: ingest(catalog="main", layer="silver", target_table="c_cliente") → main.silver.c_cliente.

Como a Execução Funciona

Use esta seção como mapa mental da lib. Ela mostra somente o fluxo que influencia operação, troubleshooting e revisão de contrato.

Entrada Contrato primeiro

O usuário declara fonte, modo, chaves, quality, schema, shape e governança. O código do notebook fica genérico.

Execução Pipeline previsível

Preparação, validação, quality e escrita rodam sempre na mesma ordem, independente da tabela.

Evidência Observabilidade nativa

Runs, erros, quality, lineage, schema changes e streaming são registrados em ctrl tables.

Contratos por responsabilidade

ingestion.yaml Engenharia

Fonte, target, modo, chaves, schema, quality, shape e performance.

annotations.yaml Catálogo

Descrição da tabela, descrições de campos, aliases, tags e PII.

operations.yaml Operação

Owner, criticidade, SLA, grupos, runbook e sinais para alertas externos.

access.yaml Segurança

Grants, row filters e masks aplicados em fluxo separado com permissão elevada.

Quick Start

from lakehouse_ingestion import ingest

df = spark.createDataFrame(
    [(1, "Alice", "2024-01-01"), (2, "Bob", "2024-01-02")],
    "id long, nome string, updated_at string"
)

result = ingest(
    source=df,
    target_table="b_clientes",
    catalog="main",
    layer="bronze",
    mode="scd0_append",
    notebook_name="hello_ingest",
    explain_mode=True,
)

print(f"Status: {result['status']}")          # SUCCESS
print(f"Linhas escritas: {result['rows_written']}")   # 2
print(f"Run ID: {result['run_id']}")           # UUID

Arquivo contracts/bronze/b_clientes.yaml:

source: raw_clientes
target_table: b_clientes
catalog: main
layer: bronze
mode: scd0_append
watermark_columns: updated_at
schema_policy: permissive
notebook_name: bronze_clientes
description: "Captura bruta de clientes do CRM"

Notebook genérico:

import yaml
from lakehouse_ingestion import ingest_plan
from lakehouse_ingestion.plan import build_plan_from_kwargs

with open("contracts/bronze/b_clientes.yaml") as f:
    cfg = yaml.safe_load(f)

plan = build_plan_from_kwargs(**cfg)
result = ingest_plan(plan)

print(f"Status: {result['status']}")

Eventos, logs, fatos transacionais. Modo padrão. Insere sem comparar com destino. Combine com watermark_columns para idempotência relativa.

    target_table="b_orders",
    catalog="main", layer="bronze",
    mode="scd0_append",
    watermark_columns="updated_at",
    source_system="erp",
)

scd0_overwrite — Substituição Total ou Parcial

Tabelas de referência, snapshots, reprocessamentos. Sobrescreve tabela inteira ou partição via partition_column + partition_value.

# Total
ingest(source=df_ref, target_table="c_status_pedido",
       mode="scd0_overwrite", schema_policy="strict")
# Parcial
ingest(source=df_mes, target_table="faturamento_mensal",
       mode="scd0_overwrite",
       partition_column="competencia", partition_value="2026-05")
source: c_faturamento
target_table: faturamento_mensal
catalog: main
layer: gold
mode: scd0_overwrite
partition_column: competencia
partition_value: "{{dt}}"      # placeholder runtime
Sem partition_value, o overwrite apaga toda a tabela. Use com cautela.

scd1_upsert — Estado Atual (SCD Tipo 1)

Modo mais comum em Silver. MERGE INTO com t.key <=> s.key (NULL-safe). UPDATE nos existentes, INSERT nos novos.

    mode="scd1_upsert", merge_keys="id_cliente",
    dedup_order_expr="updated_at DESC NULLS LAST",
    schema_policy="additive_only",
)
source: b_cliente
target_table: c_cliente
catalog: main
layer: silver
mode: scd1_upsert
merge_keys: id_cliente
dedup_order_expr: "updated_at DESC NULLS LAST"
schema_policy: additive_only

scd1_hash_diff — Append com Hash Diff

Origem sem CDC, mas você quer evitar inserir duplicatas idênticas. Calcula row_hash (SHA-256) e insere apenas linhas novas ou alteradas.

    source="b_produto", target_table="c_produto_versions",
    mode="scd1_hash_diff", hash_keys="id_produto",
    hash_exclude_columns="updated_at|source_file",
    dedup_order_expr="updated_at DESC NULLS LAST",
    partition_column="ingestion_date",
)

target_table: c_produto_versions

scd2_historical — Histórico Completo (SCD Tipo 2)

Preserva histórico com valid_from, valid_to, is_current, changed_columns. Hash calculado apenas sobre scd2_change_columns — mudanças fora delas não geram nova versão. Chaves reaparecidas geram nova versão corrente.

    source="c_cliente", target_table="dim_cliente_historico",
    mode="scd2_historical", merge_keys="id_cliente",
    scd2_change_columns="nome|email|status|cidade",
    scd2_effective_from_column="updated_at",

snapshot_soft_delete — Snapshot com Soft Delete

Snapshot completo: source = estado-fim. Linhas ausentes viram is_active=false + deleted_at=now(). Registros que reaparecem são reativados.

Não aceita watermark_columns nem filter_expression — um source filtrado marcaria dados ausentes como inativos erroneamente.
ingest(
    source="snapshot_customers_today",
    target_table="c_customer_snapshot",
    mode="snapshot_soft_delete", merge_keys="customer_id",
    dedup_order_expr="updated_at DESC NULLS LAST",
)
source: snapshot_customers_today
target_table: c_customer_snapshot
catalog: main
layer: silver
mode: snapshot_soft_delete
merge_keys: customer_id
dedup_order_expr: "updated_at DESC NULLS LAST"

Quality Gates

Estrutura do QualityRules

@dataclass(frozen=True)
class QualityRules:
    required_columns: List[str]       # Colunas OBRIGATÓRIAS
    not_null: List[str]               # Colunas sem NULL
    unique_key: List[str]             # Conjunto único
    accepted_values: Dict[str,List]   # Valores permitidos (max 1000)
    min_rows: int | None             # Mínimo de linhas
    max_null_ratio: Dict[str,float]  # Razão máxima de NULLs (0-1)
    expressions: List[QualityExpression]  # Expressões SQL

@dataclass(frozen=True)
class QualityExpression:
    name: str            # Nome único
    expression: str      # SQL booleano
    severity: "warn" | "quarantine" | "abort" = "quarantine"
    message: str | None  # Mensagem descritiva

Ações em Falha (on_quality_fail)

AçãoComportamento
"fail"Aborta a execução. Padrão.
"warn"Registra falhas, mas escreve tudo.
"quarantine"Linhas inválidas → ctrl_ingestion_quarantine; válidas → target.

Regras Abortivas (Abort-Only)

unique_key, min_rows e required_columns descrevem propriedades do conjunto — não conseguem isolar linhas. Quando falham com on_quality_fail="quarantine", o framework escala automaticamente para "fail".

Avaliação (Single-Pass)

Regras de coluna são consolidadas em uma única df.agg(...) para reduzir I/O. Exceções: unique_key (groupBy), required_columns (schema), expressions (uma passagem cada).

ingest(
    source="b_orders", target_table="c_orders",
    mode="scd1_upsert", merge_keys="order_id",
    quality_rules={
        "required_columns": ["order_id", "updated_at", "status"],
        "not_null": ["order_id", "updated_at"],
        "unique_key": ["order_id"],
        "accepted_values": {"status": ["open", "closed", "cancelled"]},
        "min_rows": 1,
        "max_null_ratio": {"customer_email": 0.20},
        "expressions": [
            {"name": "positive_amount", "expression": "amount > 0",
             "severity": "quarantine", "message": "Valor deve ser positivo."},
            {"name": "valid_period", "expression": "end_date >= start_date",
             "severity": "abort", "message": "Período inválido."},
        ],
    },
    on_quality_fail="quarantine",
)
quality_rules:
  required_columns: [order_id, updated_at, status]
  not_null: [order_id, updated_at]
  unique_key: [order_id]
  accepted_values:
    status: [open, closed, cancelled]
  min_rows: 1
  max_null_ratio:
    customer_email: 0.20
  expressions:
    - name: positive_amount
      expression: "amount > 0"
      severity: quarantine
      message: "Valor deve ser positivo."
    - name: valid_period
      expression: "end_date >= start_date"
      severity: abort
      message: "Período inválido."
on_quality_fail: quarantine

Schema Policy

PolíticaNovas ColunasRemoçõesMudança TipoUso
"permissive"✅ Adiciona✅ Aceita❌ Bloqueia insegurasBronze, origens instáveis
"additive_only"✅ Adiciona❌ Rejeita❌ Bloqueia insegurasSilver/Gold com contratos crescentes
"strict"❌ Rejeita❌ Rejeita❌ RejeitaGold, schema fixo

Alargamento de Tipo (allow_type_widening=True)

Permite: int→bigint, float→double, decimal(p,s)→decimal(p',s') com p'≥p, date→timestamp, int→double. Aplica ALTER TABLE ALTER COLUMN TYPE e registra em ctrl_ingestion_schema_changes.

allow_type_widening=True é incompatível com schema_policy="strict".
# Bronze: origem instável
schema_policy: permissive
# Silver: contrato que só cresce
schema_policy: additive_only
allow_type_widening: true   # int→bigint, float→double, etc.
# Gold: schema fixo
schema_policy: strict

Quando usar Auto Loader: prefira para landing em arquivos, alto volume, chegada recorrente ou origem que pode ser descarregada em storage. Para APIs grandes ou contínuas, grave primeiro em arquivos e use Auto Loader, em vez de consultar REST diretamente em cada ingestão.

Snapshot completo: use source.read.source_complete=true apenas quando a origem contém o estado completo necessário para o modo. snapshot_soft_delete e replace_partitions dependem dessa garantia para evitar deleções ou substituições incorretas.
Observabilidade: ctrl_ingestion_runs registra conector, provider, formato, path, opções redigidas, request/auth/pagination/incremental/limits, capabilities e source_metrics_json. Em REST, as métricas incluem requests, páginas, registros, bytes, paginação, retry/rate limit e watermark aplicado. Em JDBC, incluem estratégia de leitura, incrementalidade, watermark, particionamento e fetchsize. Em Snowflake/BigQuery, incluem o formato Spark externo usado e se a origem foi declarada como completa. Secrets no formato {{ secret:scope/key }} são resolvidos via Databricks Secrets ou variável CONTRACTFORGE_SECRET_SCOPE_KEY.
Incremental: source.incremental empurra o watermark anterior para REST (watermark_param, watermark_header, watermark_body_field) ou JDBC (watermark_column/predicate). O controle oficial continua em watermark_columns.
Extensível: novos conectores podem ser registrados com register_source_resolver("meu_conector", resolver). O YAML aceita nomes válidos de conectores customizados; a execução exige que o resolver esteja registrado no runtime.

Secrets, validação e descoberta

Use {{ secret:scope/key }} em options, request, auth, pagination, response ou limits. A resolução tenta primeiro a variável CONTRACTFORGE_SECRET_SCOPE_KEY; se não existir, usa Databricks Secrets via dbutils.secrets.get. Valores sensíveis são redigidos antes de ir para logs e ctrl tables.

Shape — Normalização de Estruturas Complexas

Disponível em v1.8+. Transformação declarativa de JSON, strings JSON, structs aninhadas e arrays em colunas tabulares — sem código PySpark repetitivo.

O Problema

# Fonte com JSON complexo:
{ "id": 1, "cliente": {"id": 99, "nome": "Alice", "endereco": {"cidade": "SP"}},
  "itens": [{"produto":"A","qtd":2}, {"produto":"B","qtd":1}],
  "metadata": {"fonte":"api_v2"} }

A Solução Declarativa (YAML)

shape:
  parse_json:                 # opcional: converte string JSON em struct/array antes do shape
    - column: payload
      schema: "STRUCT<cliente: STRUCT<id: BIGINT, nome: STRING, endereco: STRUCT<cidade: STRING>>, itens: ARRAY<STRUCT<produto: STRING, qtd: BIGINT>>, metadata: STRUCT<fonte: STRING>>"
      alias: payload_json
  columns:                    # extrai campos aninhados como colunas top-level
    payload_json.cliente.id: { alias: cliente_id }
    payload_json.cliente.nome: { alias: nome_cliente }
    payload_json.cliente.endereco.cidade: { alias: cidade }
  arrays:                     # trata arrays com múltiplos modos
    - path: payload_json.itens
      mode: size
      alias: qtd_itens
    - path: payload_json.itens
      mode: first
      alias: item_principal
  flatten:                    # achata structs recursivamente
    enabled: true
    separator: "_"
    include: ["payload_json"]
    exclude: ["payload_json.metadata.config_interna"]

String JSON antes de Shape

Quando uma API, arquivo CSV ou tabela landing entrega o payload JSON como string, use shape.parse_json. Essa etapa roda apenas dentro de shape e apenas para as colunas declaradas; sem shape, o framework não tenta converter strings automaticamente.

CampoObrigatórioDescrição
columnSimColuna string ou path string que contém JSON válido.
schemaSimDDL Spark usado por from_json. Mantém o contrato determinístico.
aliasNãoNome da coluna estruturada. Sem alias, sobrescreve a coluna original.
drop_sourceNãoQuando true e alias é diferente, remove a string original após o parse. Suportado apenas para coluna top-level.
Sem inferência por amostragem: o schema é explícito porque inferir em runtime exigiria ação Spark, poderia variar conforme o lote e mudaria contratos de produção sem revisão.

Modos de Array

ModoSQL EquivalenteCaso de Uso
keep (default)—Preserva array original
to_jsonto_json(col)Auditoria, sistemas legados
sizesize(col)Métricas, validação
firstelement_at(col, 1)Valor principal do array
explodeexplode(col)Normalização 1 linha → N linhas
explode_outerexplode_outer(col)Como explode, mas preserva vazios

Segurança e Validações

Cardinalidade em Bronze: explode/explode_outer são bloqueados em Bronze por default (muda cardinalidade e quebra watermarks). Use shape.allow_cardinality_change_on_bronze: true apenas quando intencional.
Produto Cartesiano: múltiplos explodes irmãos são detectados e exigem allow_cartesian: true explícito.

FROM ops.ctrl_ingestion_runs
WHERE run_date = current_date()
ORDER BY started_at_utc DESC;

-- Estado atual de cada tabela
SELECT target_table, last_status, watermark_value, last_success_at_utc
FROM ops.ctrl_ingestion_state;

-- Falhas de qualidade recentes
SELECT q.target_table, q.rule_name, q.failed_count, q.message
FROM ops.ctrl_ingestion_quality q
JOIN ops.ctrl_ingestion_runs r USING (run_id)
WHERE r.run_date >= current_date() - 7
ORDER BY q.failed_count DESC;

-- Limpeza de dados antigos
DELETE FROM ops.ctrl_ingestion_runs WHERE run_date < current_date() - 90;

Exemplos Completos

Bronze Captura e preservação

Use append, overwrite ou hash diff. Evite regras que alterem a semântica bruta sem intenção explícita.

Silver Normalização e qualidade

Use SCD1/SCD2, quarantine, schema evolution controlado, shape e contratos de governança.

Gold Consumo e agregação

Use overwrite, upsert ou replace partitions com schema mais rígido e métricas finais de negócio.

Como escolher o exemplo certo

NecessidadeComece porPor quê
Capturar dados brutos ou arquivos novosBronze appendPreserva eventos e mantém a ingestão simples.
Consolidar estado atual por chaveSilver SCD1Atualiza registros existentes e insere novos.
Guardar histórico de alteraçõesSilver SCD2Mantém versões com vigência e coluna is_current.
Refletir snapshot completo da origemSnapshot soft deleteInativa registros ausentes na carga atual.
Publicar tabela de consumoGold overwriteRecalcula resultado final com contrato mais rígido.

FAQ

Posso usar com Structured Streaming?

Para streaming contínuo, não. A lib suporta Auto Loader em available_now, que é uma execução finita com checkpoint e foreachBatch. Para processamento contínuo, use DLT/Lakeflow ou Structured Streaming direto.

Suporta CDC via Change Data Feed como origem?

Não nativamente. Você pode processar o CDF antes e passar um DataFrame.

on_quality_fail="quarantine" isola todas as falhas?

Não. Apenas regras de linha: not_null, accepted_values, max_null_ratio, e expressions com severity="quarantine". unique_key, min_rows e required_columns escalam para fail.

Qual diferença entre merge_keys e hash_keys?

merge_keys: usadas na cláusula ON do MERGE (scd1_upsert, scd2, snapshot). hash_keys: usadas para encontrar o "último estado" no target em scd1_hash_diff.

Posso usar partition_column + cluster_columns juntos?

Não. São mutuamente exclusivos na criação da tabela — cluster tem prioridade.

Como testar um contrato YAML sem rodar?

Adicione dry_run: true. O framework valida schema, quality, watermark sem escrever nem criar ctrl tables.

O master_run_id é igual ao {{job.run_id}} do Databricks?

Não, mas você pode usá-lo como valor. master_run_id no framework é uma STRING que aceita qualquer valor — UUID, run_id numérico, ou correlation id externo.

Como funcionam os presets? São obrigatórios?

Presets são opcionais — expandem defaults. Use preset: silver_scd1_upsert ou declare todos os campos manualmente. O usuário sempre vence o preset.

O shape funciona em qualquer camada?

Sim, mas explode/explode_outer são bloqueados em Bronze por default (mudam cardinalidade). Use allow_cardinality_change_on_bronze: true para liberar.

Diferença entre ingest() e ingest_bundle()?

ingest() executa só ingestion. ingest_bundle() carrega os 4 contratos e aplica ingestion + metadata + ownership. Access requer chamada separada com permissões elevadas.

Preciso de todos os 4 contratos YAML?

Não. O mínimo é *.ingestion.yaml. Adicione os demais conforme maturidade: annotations para catálogo, operations para SLAs, access para grants/filters/masks.

Segurança e Secrets

Secrets

Use sempre {{ secret:scope/key }}. A resolução tenta variável de ambiente e depois Databricks Secrets.

Redação

Chaves contendo password, secret, token, api_key, authorization ou key são redigidas nas ctrl tables.

Ctrl tables

Restrinja o schema ops. Quarentena e erros podem conter dados sensíveis ou stack traces completos.

Explain e lineage: não coloque credenciais ou literais sensíveis em SQL, filtros ou expressões. explain_mode e OpenLineage devem carregar metadados operacionais, não payloads confidenciais.

Referência completa: docs/seguranca.md.

Checklist Pré-Produção

  • Pacote instalado no cluster (import lakehouse_ingestion; print(__version__))
  • Schema ops existe e cluster tem CREATE TABLE
  • Permissões UC: USE CATALOG, USE SCHEMA, CREATE TABLE, MODIFY, SELECT
  • Cada contrato tem notebook_name único
  • merge_keys / hash_keys validados contra amostras reais
  • quality_rules com ao menos not_null nas chaves
  • Para SCD2: scd2_change_columns restrito às colunas de negócio
  • Para snapshot: source realmente completo (sem watermark/filter)
  • dry_run=True executado ao menos uma vez
  • dedup_order_expr definido quando há risco de múltiplas versões
  • Job/workflow com retry configurado (≥1)
  • Alerta configurado para status=FAILED
  • Plano de limpeza de ctrl tables (retenção N dias)
