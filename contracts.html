Contracts

Split responsibilities without splitting ownership of the table.

ContractForge supports a single ingestion YAML for small use cases and a four-file bundle when governance, operations and access need their own review cycles.

*.ingestion.yaml

How data is read and written

Source, target, write mode, keys, schema policy, quality gates, transformations and runtime behavior.

*.annotations.yaml

How data is described

Table and column descriptions, aliases, tags, PII and lifecycle metadata applied to the catalog when supported.

*.operations.yaml

How data is operated

Business owner, technical owner, criticality, SLA, support group, runbook and tags for dashboards.

*.access.yaml

How access is governed

Grants, row filters and column masks validated and applied by dedicated commands with appropriate privileges.

Bundle layout

contracts/
  silver/
    s_customers.ingestion.yaml
    s_customers.annotations.yaml
    s_customers.operations.yaml
    s_customers.access.yaml

Ingestion contract example

source:
  type: connector
  connector: postgres
  options:
    url: "{{ secret:crm/postgres_url }}"
    dbtable: public.customers
    driver: org.postgresql.Driver

target:
  catalog: main
  schema: crm_curated
  table: s_customers

layer: silver
mode: scd1_hash_diff
hash_keys: [customer_id]
dedup_order_expr: updated_at DESC NULLS LAST
schema_policy: additive_only
quality_rules:
  not_null: [customer_id]
  unique_key: [customer_id]

Annotations example

Use annotations for catalog-facing information: descriptions, aliases, tags, PII classification and field lifecycle. This keeps table knowledge near the ingestion contract without mixing it into read/write mechanics.

policy: warn
table:
  description: Curated customer dimension used by CRM analytics.
  aliases: [clients, account holders]
  tags:
    domain: crm
    data_product: customer360
    contains_pii: "true"
columns:
  customer_id:
    description: Stable customer business key.
    tags:
      key: "true"
      required: "true"
  email:
    description: Customer email used for contactability analysis.
    pii:
      enabled: true
      type: email
      sensitivity: restricted
  legacy_code:
    description: Legacy identifier kept for migration reconciliation.
    deprecated:
      since: "2026-05-01"
      replacement: customer_id

Access example

Access is intentionally separate because it may require elevated privileges and a different approval workflow.

mode: validate_only
on_drift: warn
grants:
  - principal: account users
    privileges: [SELECT]
  - principal: data-platform
    privileges: [SELECT, MODIFY]
column_masks:
  - column: email
    function: main.security.mask_email
    using_columns: [email]
row_filters:
  - name: region_filter
    function: main.security.filter_region
    columns: [region]

Validation workflow

contractforge validate contracts/silver/s_customers.ingestion.yaml
contractforge validate-bundle contracts/silver/s_customers
contractforge governance-check contracts/silver/s_customers

Use validation in CI before Databricks jobs run. The goal is to detect broken contracts while reviewing code, not while the cluster is already processing data.

When to keep a single file

Small internal projects can start with only *.ingestion.yaml. Split files become valuable when metadata, operations and access rules need independent reviewers or when the same table is maintained by multiple roles.

Practical rule

If changing a value should not require a data reload, it probably belongs in annotations.yaml, operations.yaml or access.yaml, not in transformation logic.
