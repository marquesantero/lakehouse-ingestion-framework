Snowflake and BigQuery

These connectors delegate to Spark connectors installed in the runtime. ContractForge validates contract shape and records metadata, but the connector library must exist.

Snowflake

source:
  type: connector
  connector: snowflake
  options:
    sfURL: "{{ secret:snowflake/url }}"
    sfDatabase: ANALYTICS
    sfSchema: PUBLIC
    sfWarehouse: LOAD_WH
    dbtable: ORDERS

BigQuery

source:
  type: connector
  connector: bigquery
  options:
    table: project.dataset.orders
    parentProject: project

Operational guidance

  • Install and validate the Spark connector before running ingestion contracts.
  • Keep provider credentials in the runtime or secret manager.
  • Use connectors doctor as a static checklist, not as a live connection test.

Compatibility checklist

  • The Spark connector package or jar must be installed in the runtime.
  • Credentials should come from secrets or runtime identity, not plaintext contracts.
  • Large reads should use provider-native partitioning/pushdown options where available.
  • Control tables still capture ContractForge run metadata even though the actual read is delegated to the external Spark connector.

How this connector fits the contract

Keep extraction concerns in source, structural normalization in transform, validation in quality_rules and target semantics in mode. This separation keeps examples portable and prevents connector-specific workarounds from becoming hidden business logic.
